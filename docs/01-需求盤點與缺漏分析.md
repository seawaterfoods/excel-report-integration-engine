## 一、需求現況總覽

### 1.1 核心功能

| # | 功能項目 | 需求狀態 | 說明 |
|---|---------|---------|------|
| 1 | 區域讀取模式 | ✅ 已定義 | 指定起始單元格，自動抓取橫向與縱向資料 |
| 2 | 橫向讀取模式 | ✅ 已定義 | 指定起始至結束單元格 (如 B4~Z4) |
| 3 | 縱向讀取模式 | ✅ 已定義 | 指定起始至結束單元格 (如 B4~B24) |
| 4 | 缺值填補 | ✅ 已定義 | 讀取匯入檔非空值，填入產出檔空白處 |
| 5 | 樣板繼承 | ✅ 已定義 | 保留範例檔格式、公式，僅填入空值 |

| 7 | 僅讀第一頁籤 | ✅ 已定義 | 以 Sheet Index 0 為準 |


### 2.2 樣板結構 (2 個工作表)

### 2.4 關鍵發現

#### 🔴 資料衝突 — File 32 與 File 46

```
32_各式準備金提存時間調查表 → B22: 'V', G22: 'V'
46_各式準備金提存時間調查表 → B22: 'V', G22: 'V'  ← 同一儲存格!
```

兩份檔案填入相同座標 (B22, G22)，依需求規格「重複填值需紀錄並停止匯出」，
此為**內建的測試案例**，可驗證重複檢核機制是否正常。

#### ⚠️ 特殊值處理

File 19 (中再) 含非標準值：`V(KPMG)`, `V(集團)`，系統須能原樣寫入，不做格式轉換。

## 三、缺漏分析

### 3.1 🔴 高優先 — 影響開發可行性

| # | 缺漏項目 | 說明 | 建議 |
|---|---------|------|------|
| 1 | **讀取模式的 YAML 配置格式未定義** | 需求提及 3 種讀取模式，但 `application.yml` 範例中僅有 import-dir/output-dir/year/month，未定義讀取範圍的設定方式 | 需設計 YAML 配置結構，定義每份報表的讀取範圍 |
| 2 | **預期匯入總數的設定位置未定義** | 需求提及需設定預期匯入總數，但未在 YAML 中出現 | 新增 `expected-file-count` 參數至 YAML |
| 3 | **多報表類型的處理規則未定義** | 目錄結構暗示同一年月可能有不同報表類型，但未說明如何區分不同報表的樣板與匯入檔 | 建議以 templet 檔名作為報表識別 key，匯入檔底線後字串需與 templet 檔名匹配 |

### 3.2 🟡 中優先 — 影響完整性

| # | 缺漏項目 | 說明 | 建議 |
|---|---------|------|------|
| 4 | **多工作表處理策略** | 需求僅提「讀取第一頁籤」，但樣板含公式工作表 (彙整結果)。產出檔是否需保留所有工作表？ | 建議：產出檔完整複製樣板所有工作表，僅對第一頁籤做資料填入 |
| 5 | **錯誤報告格式** | 需求僅說「跳出錯誤提示」、「紀錄問題」，未定義格式 | 建議：產出 JSON/文字格式的執行報告 |
| 6 | **產出檔命名的「原檔名」定義** | 規格寫「原檔名底線後的字串」，但匯入檔與樣板同名。20 份匯入檔底線後字串相同，應以樣板檔名為準 | 建議：以 templet 檔名作為產出檔名基底 |
| 7 | **Docker 部署規格** | 僅列出目錄映射，缺少 Dockerfile、docker-compose | 開發後期建立 |

### 3.3 🟢 低優先 — 可後續補充

| # | 缺漏項目 | 說明 | 建議 |
|---|---------|------|------|
| 8 | **使用介面** | 需求提及「設定介面或設定檔」，目前僅規劃 YAML 設定檔，是否需 Web UI? | 初期以 CLI + YAML 為主，後續可擴展 REST API |
| 9 | **日誌規格** | 未定義 log level、log 輸出位置 | 使用 Spring Boot 預設 Logback，支援 console + file |
| 10 | **效能指標** | 未定義處理上限 (檔案大小、數量) | 目前規模 (20檔/批) 不需特別優化 |
| 11 | **年月篩選邊界** | 4 種年月組合已定義，但缺少：無效年月、目錄不存在時的處理 | 新增邊界檢查與友善錯誤訊息 |

## 四、建議確認事項

在正式開發前，建議與需求方確認以下問題：

1. **讀取模式的配置方式**：是否每份報表都需要獨立配置讀取範圍？還是統一用「缺值填補」模式 (比較 templet 與匯入檔差異) 即可涵蓋所有情境？
2. **File 32/46 的衝突是否為刻意安排的測試資料**，以驗證重複檢核？
3. **是否需要 Web UI**，還是純粹以 CLI + 設定檔方式運行？
4. **同一月份是否可能有多種不同的報表**，還是每月僅一種報表類型？
5. **產出檔的工作表處理**：是否需要完整保留樣板的所有工作表 (含公式頁)？
